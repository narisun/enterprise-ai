8
What Actually Goes Into the LLM — Final Message Stack
LLM Input
▾
System + developer + RAG + history + user + tool_results are serialized in a precise order, token-budgeted to fit the context window, and sent to the model.
Serialized Message Structure (in order)
system
Platform prompt + app instructions + policy rules + output schema + citation requirements + response style
developer
Tool/function definitions (JSON Schema) + response format constraints + structured output instructions
context
RAG chunks (docs, schemas, tool descriptions) + few-shot examples + citations metadata
history
Conversation turns (summarized if long) + prior assistant messages + accumulated observations
user
Current user message / query / follow-up
tool_results
Accumulated tool outputs from current agent loop iteration (if any)
Token Budget Example
Window: 128K–200KSystem: ~2KTools: ~1–3KRAG: ~4–8KHistory: ~2–4KUser: variableOutput reserve: ~4K
Longer context = higher cost + latency; budget carefully
IN All assembled context from Steps 4–7  →  OUT Complete serialized message stack, tokenized and sent to the model's inference engine
Tokenized input → Transformer inference
9
Inside the LLM — Transformer Inference
Model Internals
▾
Tokens flow through embedding, N transformer layers (attention + FFN), and an output head — generating either natural language or structured tool-call JSON.
Tokenization
BPE / SentencePiece
Text → token IDs
Special tokens: [BOS], [EOS]
→
Embeddings
Token → dense vector
+ Positional encoding (RoPE/ALiBi)
Dim: 4096–12288
→
Transformer Layers (×N)
Self-Attention: Q, K, V projections → attention scores → weighted values
Multi-head: parallel attention heads capture different patterns
FFN: feed-forward network (MLP) per token
Residual + LayerNorm after each sub-layer
KV Cache: stores computed K, V for already-processed tokens
32–128 layers depending on model (7B–400B+ params)
→
Output Head
Logits (vocab scores)
Softmax → probabilities
Sampling: temp, top-k, top-p
Stop sequences, max tokens
→
Token Stream
Autoregressive generation
Streaming output (SSE)
Tool call JSON detection
Continue until stop/max
KV Cache (Inference)
Stores K,V from previous tokensAvoids re-computing attention for processed prefixPrompt prefix caching: shared across requestsDramatically reduces TTFT
This is inside the inference engine — separate from application caches
Critical Engineer Notes
Model does NOT "know" tools — tools exposed via prompt/schemaTool calls = structured output parsed by runtimeContext window is a hard limit — exceeding it truncatesTemperature=0 ≠ deterministic (floating point)Streaming: tokens sent as generated, not batched
IN Tokenized message stack  →  OUT Stream of generated tokens — either natural language response or structured tool call JSON (parsed by orchestrator, loops back to Step 6)
LLM output → Post-model guardrails + output validation
