Key Architectural Decisions — Azure-Native:

1. Entra ID as Unified Identity (No Cognito Equivalent): Unlike the AWS stack which requires Cognito federation with Entra ID, the Azure stack uses Entra ID natively for everything: user SSO, M2M service auth (Managed Identity), API gateway JWT validation, and Key Vault access. This eliminates an entire identity federation layer, reducing latency by ~200ms per auth flow and removing a class of token-mapping bugs. The platform SDK's auth.py validates Entra ID JWTs with the same RS256 logic — only the JWKS endpoint changes.

2. Container Apps over AKS: Azure Container Apps eliminates Kubernetes cluster management while providing the same serverless container experience as Fargate. KEDA autoscaling handles event-driven scaling (HTTP traffic, queue depth), and built-in Dapr sidecars replace Cloud Map service discovery. For enterprises already on K8s, the same Docker images deploy to AKS with minimal config changes — the agent code is runtime-agnostic.

3. Managed Identity Everywhere: Every Container App uses a system-assigned Managed Identity to access Azure services (Key Vault, Blob Storage, Azure PG, Azure OpenAI). Zero passwords, connection strings, or API keys in environment variables. Key Vault references in ACA secrets resolve at runtime. This is a significant security improvement over the AWS model where IAM role assumption requires additional STS calls.

4. Azure OpenAI Service as Unified LLM Layer: All LLM calls route through Azure OpenAI Service, providing a single billing surface, RBAC-based model access, built-in content safety filters, and Azure Monitor telemetry. AOAI's Global Deployment feature handles multi-region smart routing automatically (vs. manual cross-region inference profiles in Bedrock). PTU (Provisioned Throughput Units) provide guaranteed latency SLAs for production agents.

5. Azure PG Flexible Server with Built-in PgBouncer: Azure Database for PostgreSQL Flexible Server includes a built-in PgBouncer connection pooler, eliminating the need for RDS Proxy or a separate pgBouncer sidecar. Combined with pgvector for RAG embeddings, this keeps the data layer simple: one managed database for relational data, vector search, and LangGraph checkpoints.

6. Observability via Azure Monitor + Application Insights: The OTel SDK exports traces via the Azure Monitor OpenTelemetry Distro, which auto-configures Application Insights with GenAI-specific trace enrichment (LLM call latency, token counts, model versions). Log Analytics provides KQL-powered querying across all services. For enterprises on Dynatrace, the OTel → Dynatrace export path still works unchanged.
